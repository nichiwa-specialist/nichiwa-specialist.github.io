github flow

github flow は、git hub開発チームで行われているシンプルな手順です。
とても簡単なルールですが、GitHubが非常に安定した状態を保ちながら 毎日175回ものデプロイを行い、発展し続けている基盤となっています。

masterブランチのものは何であれデプロイ可能である

というのが、GitHub flowのたった一つの厳格なルールです。
常にリリース可能な、安定したmasterを維持し続けることで、着実に安定してソフトウェアを発展させることができます。
そのmasterを起点にして作業を進めます。

  1. 新しい何かに取り組む際は、説明的な名前のブランチをmasterから作成する（例: new-oauth2-scopes）
  2. 作成したブランチにローカルでコミットし、サーバー上の同じ名前のブランチにも定期的に作業内容をpushする
  3. フィードバックや助言が欲しい時、ブランチをマージしてもよいと思ったときは、 プルリクエスト を作成する
  4. 他の誰かがレビューをして機能にOKを出してくれたら、あなたはコードをmasterへマージすることができる
  5. マージをしてmasterへpushしたら、直ちにデプロイをする

このうち、gitを操作して行うものは1〜３になります。
この1〜3を GitHub Desktop を使って行う手順がチュートリアルになっています。
このチュートリアルを順番に見ていきましょう。

masterからブランチを作成する

新たな作業に取り掛かる際には、まず左上にあるブランチボタンを押して、 これから行おうとしている作業が何かを説明するような名前でbranchを作成します。
masterブランチは常に安定しているので、いつでも安全に作業を開始できます。

ブランチ上で作業する

Github上の左のペインでプロジェクトを右クリックすると、Finder(Macの場合)やexplorer(windowsの場合)、 さらには入れていればAtom(テキストエディタ)などで編集するためのメニューが表示されます。
ここから開いて編集しましょう。

こまめにコミットする

変更すると、Github Desctopにも変更内容が表示されます。随時コミットしましょう。
上部中央の表示切替で「History」モードになっていると変更内容が表示されずコミットできないので、 その場合は「n Uncommited Change」と書かれたボタンを押して変更表示モードに切り替えてください。
コミット時にはSummaryに編集内容の説明を書き、補足があればdescriptionも書きます。 ブランチは、たいていは一人か、同じ作業で密接に協力しているごく少人数にしか影響しません。
こまめに、右上に表示される「Publish」ボタンや「Sync」ボタンを押して、サーバと同期してください。

プルリクエストを作る

作業がリリースできるレベルまで完成した時や、そうでなくても意見を求めたい時など、 レビューしてもらう必要があればいつでもpull requestを作成してください。
ブランチに対して一度プルリクエストを作成すると、その後のブランチに対するコミットは 全てそのプルリクエストに含まれるようになります。
プルリクエストをmasterにマージするのは、テストとレビューが通り、マージが自動で通る場合のみです。
何か指摘があったりテストが通らない場合はブランチ上でさらに修正を加え、
マージが自動で通らない時は一旦masterからブランチへのマージを行い、
masterにマージする前に全ての問題を解決してください。
